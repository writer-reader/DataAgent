# DataAgent — Python 版 AI 数据分析师 项目方案

> 目标：用 Python 技术栈完整复刻 [oss-data-analyst](https://github.com/vercel-labs/oss-data-analyst) 的功能：
> 一个通过"探索语义层 → 构建 SQL → 执行 → 生成叙述性报告"来回答自然语言数据问题的 AI Agent。

---

## 1. 原项目功能拆解

| 模块 | 原实现（TS/Next.js） | 作用 |
|---|---|---|
| 语义层 | `src/semantic/*.yml` | 用 YAML 描述实体、字段、度量、join 关系，是 Agent 的"数据字典" |
| Agent 核心 | Vercel AI SDK `streamText` + tools | 多步工具调用循环，最多 100 步，调用 FinalizeReport 即停止 |
| 工具 1: bash | Vercel Sandbox 中执行 `cat/grep/ls` | 让 Agent 动态探索语义层文件（不把 schema 硬编码进 prompt） |
| 工具 2: ExecuteSQL | better-sqlite3 | 只读执行 SQL，返回 rows/columns/耗时 |
| 工具 3: FinalizeReport | Zod schema 校验 | 结构化输出：sql + csvResults + narrative |
| API | Next.js `/api/chat` route | 接收消息，流式返回（SSE / UI Message Stream） |
| 前端 | React + ai-elements 组件库 | 聊天界面、推理过程展示、工具调用折叠、结果表格、CSV 下载 |
| 数据库 | SQLite（init + seed 脚本） | companies / people / accounts 三张示例表 |
| 评测（可选） | evalite | Agent 回答质量评测 |

**核心设计思想**：不把数据库 schema 塞进 system prompt，而是给 Agent 一个"文件系统探索"能力，
让它像人类分析师一样先查文档（语义层 YAML）再写 SQL。schema 变更无需改代码。

---

## 2. Python 技术选型

| 层 | 选型 | 备选 | 理由 |
|---|---|---|---|
| Agent 框架 | **手写工具循环**（已定）| Pydantic AI / LangGraph | 循环逻辑只有 ~80 行，完全可控，注释写清每一步 |
| LLM | **`openai` SDK**（已定，base_url 可指向任意兼容中转） | anthropic SDK | 用户使用 OpenAI 兼容接口；换模型只改 .env |
| Web 框架 | **FastAPI** + SSE（`sse-starlette`） | Flask | 原生 async，流式响应简单 |
| 前端 | **单文件 HTML + 原生 JS**（fetch + EventSource） | Streamlit / Gradio | 零构建；若想更快出效果可用 Streamlit |
| "沙箱" | **Python 内实现受限工具**（read_file / grep / list_files，锁定在 semantic/ 目录） | Docker / subprocess | 原项目用 Vercel Sandbox 是因为要跑真 bash；我们只需读 YAML，自己实现三个只读函数更安全、零依赖 |
| 数据库 | `sqlite3`（标准库） | SQLAlchemy | 只读查询，标准库足够 |
| SQL 安全 | `sqlglot` 解析校验（只允许 SELECT） | 正则判断 | sqlglot 还能做方言转换，为将来接 PG/Snowflake 留路 |
| 语义层 | `PyYAML` | ruamel.yaml | 与原项目 YAML 格式完全兼容 |
| 数据处理 | `pandas`（结果转 CSV / 表格） | 纯 dict | 顺便支持简单统计 |
| 配置 | `pydantic-settings` + `.env` | python-dotenv | 类型安全的配置 |
| 评测（可选） | `pytest` + 自定义 eval | promptfoo | 对齐原项目 evalite |

## 4. 各模块设计

### 4.1 Agent 循环（app/agent/loop.py）— 项目心脏

对应原项目 `streamText({ tools, stopWhen })`。基于 OpenAI SDK 手写循环，伪代码：

```python
async def run_agent(messages) -> AsyncIterator[AgentEvent]:
    """流式产出事件：文本增量、工具调用、工具结果、最终报告。

    对应原项目的 streamText 循环：
    - 每轮把完整对话（含之前的工具结果）发给模型
    - 模型要么输出文本（继续/结束），要么请求调用工具
    - FinalizeReport 被调用 => 终止（原项目的 stopWhen 语义）
    """
    for step in range(MAX_STEPS):            # 原项目 stepCountIs(100)
        stream = await client.chat.completions.create(
            model=settings.model,
            messages=[{"role": "system", "content": SYSTEM_PROMPT}, *messages],
            tools=TOOL_SCHEMAS,              # OpenAI function-calling 格式
            stream=True,
        )
        # 流式消费：文本增量实时 yield；tool_calls 的分片增量拼装成完整调用
        text_parts, tool_calls = [], {}
        async for chunk in stream:
            delta = chunk.choices[0].delta
            if delta.content:
                text_parts.append(delta.content)
                yield TextDelta(delta.content)
            for tc in delta.tool_calls or []:
                merge_tool_call_chunk(tool_calls, tc)   # 按 index 拼 name/arguments

        if not tool_calls:                   # 无工具调用 => 模型说完了，结束
            yield Done(); return

        messages.append(assistant_msg(text_parts, tool_calls))
        for call in tool_calls.values():
            yield ToolCallEvent(call.name, call.arguments)
            result = await TOOL_REGISTRY.execute(call.name, call.arguments)
            yield ToolResultEvent(call.name, result)
            messages.append({"role": "tool", "tool_call_id": call.id,
                             "content": json.dumps(result, ensure_ascii=False)})
            if call.name == "FinalizeReport":   # 原项目 stopWhen 逻辑
                yield ReportEvent(result); yield Done(); return
    yield ErrorEvent("达到最大步数限制")
```

要点：
- **stopWhen 语义**：FinalizeReport 被调用 → 立即终止循环并产出报告事件
- **tool_calls 分片拼装**：OpenAI 流式下工具调用的 `arguments` 是逐字符分片到达的，必须按 `index` 累积拼装后再 `json.loads`——这是手写循环唯一的坑，注释里会重点标注
- **事件流**：所有 yield 的事件由 FastAPI 层序列化为 SSE，前端逐条消费
- **多轮对话**：messages 由前端携带完整历史（与原项目 useChat 一致）

### 4.2 System Prompt（app/agent/prompts.py）

直接翻译原项目 prompt，只改工具名（bash → explore 工具族）：

```
你是数据分析专家 AI。通过探索语义层（YAML 文件）、构建 SQLite SQL、执行并展示结果来回答问题。

## 文件结构
- semantic/catalog.yml         实体目录（描述 + 示例问题 + 字段列表）
- semantic/entities/*.yml      实体详情（SQL 表名、字段、度量、join）

## 工作流
1. 探索: read_file("catalog.yml") 浏览实体；grep("关键词") 搜索；read_file 看实体详情
2. 建 SQL: 使用实体定义中的 table 名，t0/t1 别名，LIMIT 1001
3. 执行: ExecuteSQL；出错则分析错误、修改 SQL 重试（绝不重复相同 SQL，最多 2 次）
4. 收尾: FinalizeReport(sql, csv_results, narrative)

## 准则
- 写 SQL 前必须先探索 schema，绝不猜字段名
- 叙述先给直接答案，3-6 句，说明假设与注意事项
```

### 4.3 探索工具（app/tools/explore.py）— 替代 Vercel Sandbox

原项目为了让 Agent 跑 `cat/grep/ls` 起了一整个云沙箱。Python 版直接实现三个**只读、路径锁定**的函数，能力等价、更安全：

```python
SEMANTIC_ROOT = Path("semantic").resolve()

def _safe(rel: str) -> Path:
    p = (SEMANTIC_ROOT / rel).resolve()
    if not p.is_relative_to(SEMANTIC_ROOT):   # 防路径穿越
        raise ToolError("路径越界")
    return p

def list_files(path: str = ".") -> str: ...      # 相当于 ls -R
def read_file(path: str) -> str: ...             # 相当于 cat
def grep(pattern: str, ignore_case=True) -> str: # 相当于 grep -rn semantic/
```

> 进阶选项：如果想 1:1 复刻"真 bash"体验，可以再加一个 `bash` 工具用
> `subprocess.run(cmd, cwd=semantic_dir, timeout=5)` + 命令白名单（cat/grep/ls/head/wc）。
> 初版不建议——三个纯 Python 函数已覆盖全部实际用途。

### 4.3.1 RAG 预留接口（app/retrieval/base.py）★

初版**不引入 RAG**（保持原作"Agent 自己翻文档"的风味），但探索工具不直接读文件系统，
而是通过一个 `SemanticRetriever` 协议中转——这就是将来插 RAG 的位置：

```python
class SemanticRetriever(Protocol):
    """语义层检索协议。探索工具只依赖此接口，不关心背后是全量文件还是向量库。

    设计意图：当 semantic/ 膨胀到几百个实体、无法全量 cat 时，
    换一个 VectorRetriever 实现即可在'读取前'做预筛选，工具层和 Agent 循环零改动。
    """
    def list_files(self, question: str | None = None) -> list[str]: ...
    def read_file(self, path: str) -> str: ...
    def search(self, query: str) -> str: ...     # 初版=grep；RAG 版=向量召回+grep 混合

class PassthroughRetriever:
    """初版实现：直通文件系统（等价于原项目的沙箱 cat/grep/ls）。"""
    ...

# 将来（P5）：
class VectorRetriever:
    """RAG 实现：启动时把每个实体 YAML（按实体粒度，非固定 chunk）embedding 入库。
    - list_files(question): 先向量召回 top-k 相关实体，只返回这些文件名 => Agent 视野被预筛选
    - search(query): 向量召回 + 关键词 grep 取并集，附相似度分数
    读取（read_file）永远返回完整 YAML —— 预筛选只决定'看哪些文件'，不截断文件内容。
    """
```

关键决策记录：
- **注入点在"文件可见性"而非"文件内容"**：RAG 只影响 `list_files`/`search` 返回哪些候选，
  `read_file` 永远给全文。这样 SQL 构建所需的字段/join 信息永不残缺，RAG 召回错了
  Agent 还能通过 grep 兜底自救——保持了原作探索式架构的鲁棒性。
- **embedding 粒度 = 实体文件**：一个 YAML 一个向量（用 name+description+example_questions 做
  embedding 文本），不做段落切分。实体文件天然是语义完整的单元。
- **user question 如何传入**：Agent 循环把当前用户问题存入 request 级上下文（contextvars），
  Retriever 可选读取。工具签名对模型保持不变。

### 4.4 SQL 执行工具（app/tools/execute_sql.py）

```python
def execute_sql(sql: str) -> dict:
    parsed = sqlglot.parse_one(sql, dialect="sqlite")
    if not isinstance(parsed, sqlglot.exp.Select):
        return {"ok": False, "error": "仅允许 SELECT 查询"}
    t0 = time.perf_counter()
    conn = get_readonly_conn()      # sqlite3.connect(f"file:{db}?mode=ro", uri=True)
    cur = conn.execute(sql)
    columns = [d[0] for d in cur.description]
    rows = cur.fetchmany(1001)
    return {"ok": True, "columns": columns,
            "rows": [dict(zip(columns, r)) for r in rows],
            "row_count": len(rows),
            "execution_time_ms": round((time.perf_counter()-t0)*1000, 1)}
```

比原项目更安全的两点：`mode=ro` 只读连接（原项目其实允许 INSERT/UPDATE！）+ sqlglot AST 校验。

```python

execute 就是原样返回（identity），它的意义在于**强制模型产出结构化收尾**，循环检测到它即停止。

### 4.6 API 层（app/main.py）

```python
@app.post("/api/chat")
async def chat(req: ChatRequest):
    async def gen():
        async for event in run_agent(req.messages):
            yield {"event": event.type, "data": event.model_dump_json()}
    return EventSourceResponse(gen())

app.mount("/", StaticFiles(directory="static", html=True))
```

SSE 事件协议（对应原项目 UI Message Stream 的简化版）：

| event | data | 前端行为 |
|---|---|---|
| `text` | `{delta}` | 追加到当前气泡 |
| `tool_call` | `{name, input}` | 显示折叠的"正在探索/执行 SQL"卡片 |
| `tool_result` | `{name, output}` | 填充卡片内容 |
| `report` | `{sql, csv_results, narrative}` | 渲染结果表格 + SQL 高亮 + 叙述 |
| `done` / `error` | — | 结束/报错 |

## 6. 与原项目的差异说明

| 点 | 原项目 | Python 版 | 原因 |
|---|---|---|---|
| 沙箱 | Vercel Sandbox（真云端 bash） | 进程内受限只读工具 | 需求只是读 YAML，无需真 shell；零成本零延迟 |

| SQL 安全 | 无校验（可写库！） | 只读连接 + sqlglot AST 校验 | 修复原项目安全隐患 |
| 前端 | React + 30+ 组件 | 单文件 HTML（或 Streamlit） | 聚焦 Agent 本身 |
| Slack 集成 | @slack/bolt | 不做（可用 slack-bolt for Python 后补） | 非核心 |
